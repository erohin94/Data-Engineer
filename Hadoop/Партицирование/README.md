## Партицирование в Hadoop

Партицирование — это способ разделить большую таблицу на несколько физических частей по значению колонки.

В Hadoop партицирование — это разделение таблицы на папки в HDFS по значению колонки.

Например таблица: `events`

```
user_id	 action   report_dt
1	       click	  2026-03-01
2	       buy	    2026-03-01
3	       click	  2026-03-02
```

Если таблица `partitioned by report_dt`, то данные на диске будут лежать так:

```
/warehouse/events/

    report_dt=2026-03-01/
        part-00001.parquet
        part-00002.parquet

    report_dt=2026-03-02/
        part-00001.parquet
```

То есть партиция = папка

Логический уровень для пользователя это одна таблица. Тоесть пишем: `SELECT * FROM events WHERE report_dt = '2026-03-01'` и думаем, что работаем с одной таблицей.

Физический уровень. На самом деле данные разбиты на части. Это могут быть: отдельные таблицы (в классических БД), отдельные файлы, отдельные папки.

**Как создаётся партиционированная таблица**

В Hive / Impala:

```
CREATE TABLE events (
    user_id INT,
    action STRING
)
PARTITIONED BY (report_dt DATE)
STORED AS PARQUET;
```
`report_dt` не хранится в данных файла, оно хранится в имени папки

**Как вставляются данные**

Статическая партиция

```
INSERT INTO events PARTITION (report_dt='2026-03-01')
SELECT user_id, action
FROM source_table;
```
Файл попадёт в папку: `report_dt=2026-03-01`

Динамическая партиция

```
INSERT INTO events PARTITION (report_dt)
SELECT user_id, action, report_dt
FROM source_table;
```

Impala сама создаст папки:

```
report_dt=2026-03-01
report_dt=2026-03-02
report_dt=2026-03-03
```

**Как работает запрос**

Запрос:

```
SELECT *
FROM events
WHERE report_dt = '2026-03-01'
```
Impala делает: `partition pruning`. То есть читает только: `report_dt=2026-03-01`

Партиции ускоряют запросы. Представь таблицу: 2 млрд строк, 365 дней

Без партиций: scan = 2 млрд строк

С партициями: scan = 1 день ≈ 5 млн строк. Разница может быть 100-1000x.

## Размер партиций

**Маленький размер партиций в Impala** - это распространенная проблема, которая может вызывать:

-Медленные запросы (overhead от обработки многих маленьких файлов)

-Неэффективное использование памяти

-Избыточное потребление ресурсов

**Признаки маленьких партиций:**

-Размер партиции < 128MB (рекомендуемый минимум)

-Количество файлов >> количества партиций

-Размер многих партиций < 10MB

Посмотреть статистику таблицы можно командой `SHOW TABLE STATS database_name.table_name`. Перед использованием команды обновить статистику `COMPUTE STATS database_name.table_nam`.
